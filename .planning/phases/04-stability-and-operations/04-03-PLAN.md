2. Add the shutdown coroutine as a module-level async function (before `main()`):
```python
async def _graceful_shutdown(loop: asyncio.AbstractEventLoop) -> None:
    """Cancel all tasks, wait up to 10s, then stop the loop (D-32)."""
    logger = structlog.get_logger()
    tasks = [t for t in asyncio.all_tasks(loop) if t is not asyncio.current_task()]
    logger.info("graceful_shutdown_cancelling", task_count=len(tasks))
    for task in tasks:
        task.cancel()
    if tasks:
        done, pending = await asyncio.wait(tasks, timeout=10)
        if pending:
            logger.warning("graceful_shutdown_timeout", pending_count=len(pending))
    logger.info("graceful_shutdown_complete")
    loop.stop()
```

3. In `main()`, AFTER `loop = asyncio.get_event_loop()` (line 121) and BEFORE handler wiring, add signal registration:
```python
# SIGTERM/SIGINT graceful shutdown (STAB-02, D-31)
_shutdown_requested = False

def _handle_signal():
    nonlocal _shutdown_requested
    if _shutdown_requested:
        return  # Prevent double-shutdown
    _shutdown_requested = True
    logger.info("shutdown_signal_received")
    loop.create_task(_graceful_shutdown(loop))

loop.add_signal_handler(signal.SIGTERM, _handle_signal)
loop.add_signal_handler(signal.SIGINT, _handle_signal)
```

CRITICAL NOTES:
- `_handle_signal` is a plain callable, NOT a coroutine — `add_signal_handler` requires this (Pitfall 1 from research)
- `_shutdown_requested` flag prevents double-triggering if user hits Ctrl-C twice
- `loop.create_task()` schedules the async shutdown on the event loop
- `loop.stop()` in `_graceful_shutdown` interrupts `ws_client.start()` blocking call
- `asyncio.all_tasks()` returns ALL tasks including the cleanup task from Plan 01 and any active worker tasks from Plan 02 — all get cancelled
- Filter out `asyncio.current_task()` to avoid self-cancellation (Pitfall from research)
- `asyncio.wait(timeout=10)` gives tasks up to 10 seconds to handle CancelledError — workers from Plan 02 will finalize their cards in the CancelledError handler

**feishu-bridge.service** — Create systemd user service unit file in the project root (per D-33, D-34):

Create the file at `/home/mi/feishu-bridge/feishu-bridge.service`:
```ini
[Unit]
Description=Feishu Bridge - Claude Code bot
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=/home/mi/feishu-bridge
ExecStart=/usr/bin/python3 /home/mi/feishu-bridge/main.py
EnvironmentFile=/home/mi/feishu-bridge/.env
Restart=on-failure
RestartSec=3
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=default.target
```

NOTES:
- ExecStart uses `/usr/bin/python3` (confirmed by `which python3` in research)
- EnvironmentFile loads `.env` for APP_ID, APP_SECRET, LOG_FORMAT=JSON, SESSION_TTL, etc.
- systemd EnvironmentFile does NOT support `export` prefix or shell expansion — the `.env` file must use simple `KEY=VALUE` format (Pitfall 5 from research). If the project's `.env` has `export` prefixes, they must be removed for systemd compatibility.
- `Restart=on-failure` + `RestartSec=3` per D-34
- `StandardOutput=journal` means logs go to journald, viewable with `journalctl --user -u feishu-bridge`
- `Linger=yes` already confirmed on this machine — no `loginctl enable-linger` needed
- The file is stored in project root for version control. Deployment instructions in the summary will explain how to symlink/copy to `~/.config/systemd/user/`.
